
# OIDC, Federación de Identidad y SSO - Teoría

## ¿Qué es OIDC (OpenID Connect)?

OpenID Connect (OIDC) es un protocolo de autenticación estándar construido sobre OAuth 2.0 que permite:
- **Autenticación federada**: delegar login a un proveedor de identidad (Keycloak, Entra ID, Google, etc.)
- **Identidad verificada**: obtener información del usuario (email, roles, grupos) mediante tokens seguros
- **Single Sign-On (SSO)**: un login centralizado para múltiples aplicaciones
- **Autorización por roles**: validar permisos en backend usando claims del token

### ¿Para qué sirve en el bootcamp?

- Desacoplar autenticación de aplicaciones cliente (frontend)
- Centralizar control de identidades y roles en un proveedor (Keycloak, Entra ID, etc.)
- Implementar acceso federado sin guardar passwords en la aplicación
- Validar escenarios de seguridad: autenticación (401), autorización (403), acceso permitido (200)
- Preparar arquitectura enterprise con SSO multiaplicación

## Palabras clave

| Término | Descripción |
|---------|------------|
| **OIDC** | OpenID Connect, protocolo de autenticación sobre OAuth 2.0 |
| **OAuth 2.0** | Framework de autorización, base de OIDC |
| **Identity Provider (IdP)** | Servidor que verifica identidades (Keycloak, Entra ID, Google) |
| **Authorization Code Grant** | Flujo seguro para SPAs: app → IdP → code → backend exchange → token |
| **PKCE** | Proof Key for Code Exchange, capa de seguridad para SPAs sin backend secret |
| **ID Token** | JWT que contiene identidad del usuario (email, sub, roles) |
| **Access Token** | JWT que autoriza acceso a recursos (APIs protegidas) |
| **JWT (JSON Web Token)** | Token firmado en formato JSON (header.payload.signature) |
| **Claims** | Afirmaciones en el JWT: sub (subject), email, roles, groups, etc. |
| **Issuer** | URL del IdP que emitió el token (ej: `https://idp.company.com/realms/bootcamp`) |
| **Audience** | API/recurso para el que es válido el token (ej: `api-backend`, `mobile-app`) |
| **Realm** | Contexto aislado de usuarios, roles y aplicaciones en IdP (Keycloak) |
| **Scope** | Permisos solicitados al IdP (`openid`, `profile`, `email`, `roles`) |
| **Redirect URI** | URL a donde IdP redirige tras autenticación (ej: `http://localhost:4200/callback`) |
| **SSO** | Single Sign-On, login único centralizado para múltiples apps |
| **MFA / 2FA** | Autenticación multifactorial, seguridad adicional |
| **Federation** | Delegación de identidad entre múltiples IdPs (ej: Google + Keycloak) |
| **Bearer Token** | Token enviado en header `Authorization: Bearer <TOKEN>` |
| **JwtBearer Scheme** | Middleware que valida Bearer tokens en ASP.NET / Express / FastAPI |
| **Policy-Based Authorization** | Autorización granular mediante políticas (ej: `[Authorize(Policy="CanManageUsers")]`) |

## Flujos principales

### Authorization Code + PKCE (para SPAs)

**Seguro para aplicaciones frontend sin backend secreto:**

```
1. Usuario hace click en "Login"
2. Frontend genera PKCE code_verifier + code_challenge
3. Frontend redirige a: IdP?client_id=...&redirect_uri=...&code_challenge=...
4. Usuario ingresa credenciales en IdP
5. IdP redirige a frontend con ?code=...
6. Frontend intercambia code + code_verifier por tokens con backend (o directamente si configurado)
7. Frontend almacena access_token + id_token
8. Cada request incluye: Authorization: Bearer <access_token>
9. Backend valida token contra IdP public key
```

### Client Credentials (para backend-to-backend)

```
1. Backend A solicita token a IdP con client_id + client_secret
2. IdP valida y retorna access_token
3. Backend A usa token para comunicarse con Backend B
```

## Casos de uso

### Startup con múltiples microservicios
- Keycloak como IdP centralizado
- Frontend (Angular, React) hace login en Keycloak
- Backend API (.NET, FastAPI, Node) valida tokens de Keycloak
- Cada servicio protegido con JwtBearer

### Migración desde autenticación local a federada
- Fase 1: Ambas autenticaciones (local + OIDC) activas usando `DynamicJwt`
- Fase 2: Detectar issuer y seleccionar esquema
- Fase 3: Migración completa a OIDC, deprecar local

### SSO empresarial con Entra ID / Azure AD
- Microsoft Entra ID como IdP corporativo
- Aplicaciones internas usan Entra para login
- Sincronización de roles/permisos desde tenant Azure

### App móvil + web + backend compartido
- Frontend web (Angular) → PKCE flow → Keycloak
- App móvil (React Native) → Native flow (AppAuth) → Keycloak
- Backend (.NET, Node) → Valida tokens de ambas plataformas
